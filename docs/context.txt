StrideMarket is an educational niche marketplace application focused on running gear. The application is structured as a monorepo with separate apps and packages for the web frontend, API backend, worker processes, shared schemas, and database access. The goal of the project is to model a realistic marketplace while keeping the domain bounded enough to learn authentication, search, listing management, seller flows, media uploads, moderation, background jobs, and admin operations without turning the project into an oversized platform.

The main technologies used are Next.js, React, TypeScript, Material UI, Apollo Client, tRPC, TanStack Query, Express, GraphQL, Prisma, PostgreSQL, iron-session, bcrypt, Zod, OpenAI integration, and a database-backed background job system. The frontend lives in the web app and uses Next.js App Router with client and server components. Material UI is used for the interface, layout, forms, tables, buttons, chips, and theme system. Apollo Client is used for public marketplace reads through GraphQL, while tRPC is used for typed application workflows such as authenticated dashboards, seller listing creation, favorites, AI assistance, and admin operations.

The backend lives in the API app and uses Express as the HTTP server. It exposes REST routes for authentication and uploads, GraphQL for public marketplace reads, and tRPC for typed frontend-to-backend procedures. The API builds request context from the encrypted browser session and Prisma client, then applies role-based access checks through protected procedures and role procedures. The backend uses Zod schemas from the shared package to validate inputs consistently between client and server.

The database layer uses Prisma with PostgreSQL. The schema includes users, seller profiles, listings, listing images, categories, favorites, inquiries, reports, homepage banners, moderation logs, chat sessions, chat messages, background jobs, and email outbox records. The database package exports a shared Prisma client so API services, workers, and seed scripts use the same data access layer. Seed data creates demo accounts for admin, seller, and buyer users, creates running gear categories, creates a sample approved listing, and creates a homepage banner.

Authentication uses email and password login with bcrypt password hashing and encrypted HTTP-only cookies through iron-session. Users can register as regular users or sellers, but admins are never created through public signup. Admin accounts are seed-only or backend-controlled. The seeded admin account is admin@stridemarket.local with the seed password. Login writes the user id, email, and role into the encrypted session cookie. The application supports roles USER, SELLER, and ADMIN, with role hierarchy enforced in shared role helpers and tRPC middleware.

Regular users can browse the homepage, search listings, view categories, open listing detail pages, sign up, sign in, favorite listings, and contact sellers through the current UI surface. Regular users can also become sellers from the sell page. When a signed-in regular user visits the sell page, the application shows an option to enable seller tools on the same account. This promotes the current user to the seller role and creates or reuses the related seller profile while preserving admin-only role rules.

Sellers can create listings through the sell page. The seller listing form supports title, description, price, condition, category id, city, state, tags, AI listing assistance, and product image upload. The AI listing assistant can improve title, description, and tags using the backend AI service. When a seller submits a listing, it is created in pending review status and pending moderation state. The backend enqueues AI moderation and AI embedding background jobs. Listings are not automatically published by AI; they still require admin moderation before going live.

Seller image upload is implemented through the API uploads route. The frontend sends multipart form data to the API, the API validates file type and size with multer, saves local files to the configured storage path, returns upload URLs, and enqueues image processing jobs. The API serves local uploads statically from the uploads path. The frontend resolves media URLs so uploaded images using /uploads paths load from the API origin, while public placeholder assets continue to load from the Next.js app. Listing cards and listing detail pages use this media URL resolution.

The public marketplace features include the homepage, category browsing, search, featured listings, trending listings, listing cards, listing details, seller summary display, and homepage banners. The homepage shows a hero section, AI discovery/search component, categories, featured listings, and trending listings. Listing cards show image, category, title, price, and location. Listing detail pages show the main listing image, category, title, price, location, condition, tags, contact seller action, description, and seller name.

Search and marketplace reads are exposed through GraphQL. Public GraphQL queries include health, categories, listing by slug, search listings, featured listings, trending listings, and homepage banners. Listing search supports query text, category slug, city, state, condition, min and max price, sort order, page, and limit. Search only returns active listings that have approved moderation. Featured and trending listing queries also only return active and approved listings.

Favorites are implemented through tRPC. Signed-in users can list their favorites and toggle favorite state for a listing. The backend maintains a unique favorite relationship per user and listing and increments or decrements the listing favorite count when the user toggles favorite state. The favorites page uses this authenticated tRPC surface.

Admin functionality is implemented for the seeded admin user. Admin pages live under the /admin route group and are protected in the UI by checking the current session role through /auth/me. If the current user is not signed in as an admin, the admin pages show an admin access required state and direct the user to sign in. Admin APIs are protected on the backend with roleProcedure("ADMIN"), so the UI guard is not the only protection.

The admin overview page shows operational counts and recent listings. It displays counts for pending listings, flagged listings, active listings, open reports, users, sellers, failed jobs, and active banners. It also shows a compact recent listings table with listing details, seller details, status, and moderation state.

The admin listings page includes a moderation queue and an all listings table. The moderation queue includes listings that are pending, flagged, or pending review. Admins can approve listings, reject listings with a moderation note, and toggle whether listings are featured. Approving a listing sets the listing status to active, sets moderation to approved, clears the moderation note, sets the published date, creates an admin moderation log, and queues a listing approved email to the seller. Rejecting a listing sets status to rejected, sets moderation to rejected, stores the moderation note, creates an admin moderation log, and queues a listing rejected email to the seller.

The admin users page lists users with name, email, role, account status, and seller profile information. Admins can promote regular users to sellers, promote non-admin users to admins, suspend users, and reactivate suspended users. Promoting a user to seller creates a seller profile when needed. Admin creation is still not exposed through public signup; admin promotion is only available inside the admin-only backend procedures.

The admin banners page manages homepage banners. Admins can create banners with title, subtitle, image URL, link URL, sort order, and active state. Admins can also toggle whether existing banners are active and adjust sort order. Public homepage banner queries only return active banners ordered by sort order.

The admin reports page lists user-submitted reports. Reports are connected to reporters and optionally listings. Admins can view report reason, details, reporter information, and the reported listing when it still exists. Admins can resolve reports and reopen resolved reports.

The admin jobs page monitors background jobs. It shows job counts grouped by status and a table of recent background jobs including queue name, status, attempts, and error details. The jobs page refreshes periodically so admins can observe worker activity such as email jobs, AI moderation jobs, AI embedding jobs, image processing jobs, and trending recalculation jobs.

Background processing is handled by the worker app. The worker polls database-backed BackgroundJob and EmailOutbox records and dispatches processors for email, AI moderation, AI embeddings, image processing, and trending recalculation. The current implementation models production queue patterns while keeping the infrastructure educational and database-backed. The code is structured so a production queue system such as BullMQ and Redis could replace the simplified job table approach later.

AI features are implemented in both synchronous and asynchronous flows. Sellers can use the AI listing assistant to improve listing content. Users can use natural language search and chat-oriented AI features through the AI service. Listing creation enqueues AI moderation and AI embedding work. The moderation worker can flag content and create moderation logs, but AI does not auto-approve listings. The application keeps a human admin in the final approval path.

Media handling currently supports local development storage. Uploaded listing images are stored as files, and listing records store URLs rather than binary blobs. The media upload system validates client intent through file input accept attributes and enforces backend MIME type and file size checks through multer. The worker has a stub path for future image processing, resizing, thumbnail generation, and S3 or CDN integration.

The project uses a shared package for constants, roles, session types, and schemas. Shared Zod schemas define auth inputs, listing inputs, search inputs, and AI listing assistance inputs. This reduces duplication between frontend form validation and backend validation. The shared role helpers define the role hierarchy used by protected backend procedures.

The application approach is incremental and architectural. Public marketplace reads use GraphQL because they fit feed and detail-page data requirements. Authenticated workflows and admin actions use tRPC because it gives end-to-end type safety between the API and Next.js client. REST is used for authentication and uploads because those flows map naturally to HTTP routes and multipart form uploads. Prisma services keep business behavior close to database operations, while frontend pages stay focused on UI states and user actions.

The current implemented user-facing feature set includes registration, login, session-based auth, role-aware sell page states, buyer-to-seller upgrade, listing creation, AI listing assistance, image upload for listing creation, public homepage, public categories, public search, public listing detail pages, favorites, contact seller button surface, seeded demo data, homepage banners, media placeholder assets, and admin-only operational pages.

The current implemented admin feature set includes admin access guard, dashboard counts, recent listings, moderation queue, approve listing, reject listing, moderation logs for admin decisions, seller notification emails for listing decisions, featured listing toggles, user list, seller promotion, admin promotion, user suspension and reactivation, banner creation, banner activation toggle, banner sort order management, report inbox, report resolution and reopening, background job monitoring, and admin documentation.

The current implemented seller feature set includes seller registration, seller role authorization, seller profile creation, seller listing form, AI listing content assistance, product image upload, pending review listing submission, background job creation for moderation and embeddings, and seller access for admins through the role hierarchy.

The current implemented regular user feature set includes browsing, searching, viewing listings, account creation, sign in, favorites, contact seller entry point, and enabling seller tools from an existing account. Regular users cannot create listings until they become sellers, cannot access admin pages, and cannot perform moderation or platform management actions.

The current security model relies on server-side role checks for sensitive operations. The frontend shows role-appropriate screens, but privileged behavior is enforced by backend tRPC procedures. Admin actions require ADMIN role, seller listing creation requires SELLER role or higher, and favorites require an authenticated session. Admin accounts are intentionally excluded from public signup.

The project documentation is organized by architecture, getting started, authentication, database, GraphQL and tRPC, AI features, workers, frontend, media uploads, admin and RBAC, and learning roadmap. This context file summarizes the application state up to the current implementation so a future developer or AI assistant can understand the technologies, architecture, roles, and implemented feature surface quickly.
